安装nRF-Connect-SDK

发表于2022-12-04|更新于2022-12-23|NordicNCS
|阅读量:|评论数:

​ nRF Connect SDK，简称NCS，是Nordic最新的SDK平台，该平台将支持Nordic所有产品线，包括低功耗蓝牙，蜂窝网，WiFi，GPS，2.4G，蓝牙Mesh，Zigbee，Thread，Matter, Homekit, FindMy等。

​ NCS 内嵌Zephyr RTOS，并沿用了Zephyr project的编译系统。利用Device Tree和Kconfig进行项目的硬件、软件配置。使用CMake和大量Python脚本辅助生成一些头文件、代码和Hex。一旦上手，开发调试起来非常方便，并且是跨平台的（Windows/Linux/OSX）。

​ NCS在Github上托管，包含多个仓库。其主仓库（Manifest）是nrf（含Nordic产品驱动与各类无线协议栈等），此外还有Zephyr、MCUBoot、mbedtls、nrfxlib等其他仓库。

​ 更多信息可参考：

  • About the nRF Connect SDK — nRF Connect SDK 2.1.2 documentation (nordicsemi.com)

  • 开发你的第一个nRF Connect SDK(NCS)/Zephyr应用程序 - iini - 博客园 (cnblogs.com)

1. 简介

​ 本节将会详细介绍如何在一台Windows 11的电脑上安装NCS开发环境（Windows 10也适用），包含NCS、编译器以及其他工具。如果你已经安装好NCS开发环境，可以跳过本节。

​ 完整的中文教程，也可参考这篇文章：开发你的第一个nRF Connect SDK(NCS)/Zephyr应用程序 - iini - 博客园 (cnblogs.com)

​ 也可参考官方安装教程：Getting started — nRF Connect SDK 2.1.2 documentation (nordicsemi.com)

​ 需要安装的内容列表：

序号 软件 用途
1 Visual Studio Code 代码文本编辑器，并且通过安装插件的方式为其他开发调试工具提供可视化界面
2 nRF Command Line Tools Jlink驱动、nrfjprog等命令行工具和DLL
3 nRF Connect for Desktop 桌面工具集合，含SDK安装器、功耗评估、射频抓包等工具
4 Git 版本管理工具
5 Python 3.9 为编译套件的python脚本提供运行环境
6 Ninja 快速构建工具（类似make）
7 CMake 根据配置生成.ninja或Makefile文件的工具
8 Zephyr SDK 交叉编译工具链（编译器、链接器等）
9 NCS 2.1.2 含nRF产品驱动包、内核、第三方库等代码和编译配置文件等
10 VS Code: nRF Connect插件包 在VS Code中为NCS功能提供可视化

3. 安装SDK和工具链

3.1. 自动安装与手动安装的区别

​ 列表中4~10号软件有2种安装方式，自动安装和手动安装。

​ 自动安装是通过刚才已经装好的3号软件 nRF Connect for Desktop 中的工具安装的，这种安装方式会从GitHub拉取NCS仓库的一个特定版本（如v2.1.2），然后会把列表中剩余的软件工具（4~8号）放在toolchains文件夹下，与NCS的目录为并列关系。

​ 自动安装可以同时安装多个版本的NCS，并在同一个toolchains文件夹下安装对应版本适合的工具，如下图：

toolchains下仍有两个NCS单独的工具链

​ NCS大约有2.7GB，而toolchains大约有1.1GB。由于是从GitHub拉取，在国内如果没有网络代理基本很难下载成功，或者下载很慢。

自动安装与手动安装的其他区别：

​ 自动安装的工具不会添加到Windows全局的PATH环境变量中，因此打开VS Code后需要在nRF Connect插件的设置中选择工具链的路径。并且，随便打开一个终端（Cmd/Powershell/Git bash）是无法使用toolchains中的工具的。

注：全部安装完毕后，可以重启电脑，以应用新的环境变量

总结

​ 你可以使用 VS Code 提供的 Multi-Root 工作区来同时管理多个项目文件夹。其中每个项目文件夹在 VS Code 中都可以使用"name"参数来索引。

​ VS Code有许多设置项（Settings），VS Code安装的插件也会带有很多设置项。可以在全局（User）、工作区（Workspace）、文件夹（Folder）级别进行设置，后者会覆盖前者的设置。

  • 全局设置保存在%APPDATA%\Code\User\settings.json
  • 工作区设置保存在工作区xxxx.code-workspace文件中
  • 文件夹设置保存在此文件夹的.vscode/settings.json中

​ nrf-connect插件的APPLICATIONS表中列举的Application，都是Settings中nrf-connect.applications中配置的文件夹。这里的文件夹路径可以使用 VS Code 变量${workspaceFolder:NAME}。

​ 如果你要把一个项目从工作区中移除，除了要从workspace的folders中移除外，还要在nrf-connect插件的applications中将其移除。

5. 创建一个新工程并用git管理

​ 上一节讲解了如何打开一个或多个例程。

​ 如果我们只是打开例程，例程的文件夹还是在ncs仓库内部，受到ncs的git仓库的管理。如果想自己开发项目，并用git管理版本，就需要创建新工程。

​ NCS支持把例程当作模板，复制到NCS外部，并创建新工程。

5.1. 创建新工程

​ NCS支持以例程作为模板，复制并创建新的工程。

​ 在 VS Code中，选择左侧nRF Connect for VS Code插件，进入Welcome页面，选择 Create a new application：

  • 需要设置项目的名称、路径。
  • 需要从ncs的例程中选择一个作为模板（可以是nrf/sample/, nrf/application/, zephyr/sample/, zephyr/application/目录下的例程）
  • 需要设置Application Type

理解Application Type:

新项目的创建方式有freestanding和workspace两种：

  • freestanding 就是不使用工作区，只使用文件夹来保存一个单独的工程。这种方式，要求电脑上已经装了一个NCS。
  • workspace就是在创建这个项目时，同时创建一个工作区，并且要设定这个工作区的名称。这种方式不需要你已经安装一个NCS，因为它要直接从github上拉取一个新的NCS。

​ 如果创建新工程时，当前VS Code已经打开了一个workspace （UNTITLED临时的也算），那么就只能选择freestanding，不能选择workspace创建方式。并且这个freestanding的工程会强行加入到目前已经打开的workspace中。如果想避免这种情况，可以在VS Code菜单栏的File…选项中选择关闭当前workspace，或者打开一个新VS Code窗口，再进行操作。

​ 如果你从没用过git，需要先配置用户名和邮箱。这个用户名和邮箱不是登陆什么网站用的，而是一个签名，在提交代码时用于标记这段代码是谁提交的。这个配置存在你电脑的本地，并且是全局的，对所有git仓库都有效。

6. 编译工程

​ 不论是打开的示例工程，还是创建的新工程，都遵循下列步骤进行编译和烧录等工作。

6.1. 创建一个Build

​ 所谓Build的配置，就是一个项目要如何编译、有哪些编译参数。比如同一个项目，你可以创建一个Debug Build和一个Release Build，采用不同的优化等级。

​ 创建Build，需要选择自己使用的板卡，Zephyr自带Nordic官方的开发板，和一些其他厂商的官方开发板。Zephyr也支持使用自定义的板子，但是本文不介绍如何自定义板子。

​ 下图中，选择板子时，有三个选项：

  • Compatible boards：本例程适配的板子，如果选择这些板子，不需要任何修改就可以烧录进去使用
  • Nordic boards：Nordic 的官方开发板，以及一些Nordic的demo板
  • All boards：Zephyr 支持的所有开发板，含其他厂商的开发板

Nordic开发板board配置的命名规则：

​ 例如：nrf52840dk_nrf52840，是说这个板子的配置是为 nrf52840dk 这块开发板上的 nrf52840 这颗 MCU 创建的，会记录这个MCU的外设地址，以及此MCU连接的外部硬件的信息（如SPI Flash）。

​ 例如：nrf9160dk_nrf9160和nrf9160dk_nrf52840，都是nrf9160dk这块开发板的配置。但是这块开发板上有两颗控制器，一颗是9160 SiP，另一颗是52840。所以有两个配置可选，分别为这两颗控制器编译固件。

​ 例如：nrf5340dk_nrf5340_cpuapp和nrf5340dk_nrf5340_cpunet，都是nrf5340dk这块板子的配置，并且这块板子上只有nRF5340这一颗主控。但是nRF5340是一颗双核MCU，所以，可以有两种配置来区分。前者记录应用核的寄存器地址、外部连接的硬件信息，后者记录网络核的寄存器地址、外部连接的硬件信息等。

​ 例如：nrf5340dk_nrf5340_cpuapp和nrf5340dk_nrf5340_cpuapp_ns，都是nrf5340dk开发板上，nrf5340芯片的应用核的配置。但是，这颗应用核使用的CPU是Cortex-M33，基于Arm V8架构，提供了TrustZone的安全保护技术，同样的一个外设寄存器，可以有安全（Secure）和非安全（Non-Secure）两个地址，这样可以把安全应用和非安全应用隔离开来。因此，这两个board配置的不同之处，就是从安全地址还是非安全地址去访问芯片上的外设资源。

​ 例如：nrf52833dk_nrf52820。这块开发板上只有nrf52833这一块主控。但是由于nRF52833和nRF52820同属nRF52系列，52820上的资源是52833的子集，并且Nordic并未单独为52820制作开发板，因此可以用52833来模拟52820。此配置文件限制了52833上的硬件资源，使其表现和52820相同。

​ 所有的board信息，都可以在官网找到：Board support

9. 更新SDK

​ 如果使用自动安装的方式，可以用nRF Connect for Desktop，直接安装一个全新的SDK即可。如果想在手动安装的SDK进行升级，遵循以下步骤：

（1） 确保仓库是clean状态

​ 升级前，需确保仓库是clean的。假如，我之前更改了SDK中一个例程的配置，那么仓库就不是clean的，用git status命令查看仓库的状态：

​ git 会提示我们，哪些文件被修改了。比如这里提示我们samples/bluetooth/peripheral_uart/prj.conf被修改过。你可以有以下几种选择：

  • git stash把这些修改缓存在本地，升级完毕后，再git stash pop释放出来。本文不详细讲解git stash。
  • 直接把想保存的文件拷贝到其他地方，保存。然后按照git的指示，git restore samples/bluetooth/peripheral_uart/prj.conf 来丢弃之前的修改，从而把仓库还原到初始状态。
  • 不想保存了，直接git restore samples/bluetooth/peripheral_uart/prj.conf 丢弃之前的修改。

操作完毕后，确保仓库是clean的：

​ 提示work tree clean即说明仓库已经是clean的。
